# 代理

到目前为止，我们知道 AI 模型可以调用工具来扩展能力。那如果让它们调用多个工具、自己决定用哪些工具，甚至从结果中学习呢？这就是**代理**的作用。

从本质上说，代理就是**工具在循环中运行**。

可以这样理解：你无需一步步告诉 AI 怎么做，而是给它一个目标，让它自行规划步骤。就像与其提供逐步导航，不如只告诉对方目的地，让他们用 GPS 自行到达。

## 代理在实际中的工作方式

假设你让一个 AI 代理"在我的设置页面添加一个深色模式开关"。代理开始行动，首先在你的代码库中搜索定位设置页面。找到相关文件后，它会读取这些文件，以理解当前的结构和样式方案。

接下来有趣的一幕发生了：代理会为实现该功能自行制定计划。它可能决定添加一个状态变量、创建新的 CSS 类、实现开关组件，并更新 UI。在执行每一步时，它会持续自检、运行测试，并修复出现的任何错误。

整个过程通过一系列工具调用完成，代理会根据每次操作的结果决定下一步要做什么。这就像在旁听一个人把思路说出来，只不过它一边思考一边把工作做完。

### 📝 小测验：哪些任务通常适合委托给代理？（多选）

- ✅ **在多文件中进行模式化重构** — 代理可以按清晰的模式重构代码。
- ✅ **为一个有明确范围的失败用例添加测试** — 代理可以运行测试来验证更改。
- ❌ **基于复杂 Figma 原型的像素级重设** — 这通常需要更多人工审查和迭代。
- ❌ **集成全新且无文档的库** — 存在高幻觉风险，且可能产生不正确的代码。

## Agent 正在改变我们的构建方式

Agent 的真正魔力在于，它们正在重塑你作为开发者的角色。

与其进行"components 文件夹里有哪些文件？"、"能读一下 Button 组件吗？"、"现在把它更新为支持深色模式变体"这样的来回对话，不如直接说明你的最终目标，把过程交给 agent 处理。

这带来了巨大的转变——你将成为**任务的管理者，而非执行者**。你可以让多个 agent 同时处理代码库的不同部分：当一个 agent 为认证流程补充测试时，另一个可以更新文档，第三个则在后台重构那个你一直想清理的杂乱工具文件。

## 与代理协作的现实

代理在目标清晰、模式成熟的任务上表现出色，例如为现有代码添加测试、更新文档、按一致模式进行重构，以及修复具有明确报错信息的缺陷。

然而，对于需要深入理解系统交互的复杂调试、将视觉设计稿精确到像素的还原、使用其训练数据中未包含的新库等任务，它们仍会遇到困难。

可以把代理看作需要明确指令的高速"初级开发者"；它们也很容易遗忘，因此需要监督。它们可能陷入循环，反复采用同样失败的方法而没有意识到需要更换策略。

由于包含大量工具调用和迭代，代理消耗的 token 远多于回答简单问题。缺乏良好约束时，它们可能热情地做出你并不打算的更改。至关重要的是，你仍需负责验证代码能否正确运行并符合你的标准。

### 📝 小测验：哪项护栏有助于防止代理进行无限制的更改？

- ❌ **禁用所有工具以保证代理安全** — 这是移除能力，而不是进行约束。
- ✅ **在合并更改前要求通过测试** — 虽然并非完美方案，但这是一个有用的护栏。
- ❌ **让代理"多加小心"** — 这虽然有助于提示，但并不能防止代理进行更改。

## 委派的艺术

与代理高效协作，关键在于学会什么该委派、何时委派。先从小而明确的任务开始，建立信心。随着逐渐熟悉，你可以委派更大块的工作，但务必在过程中设置检查点和验证步骤。我们将在课程后续介绍更多实用示例。

目标不是消除人工参与，而是放大你的成效。你担任架构师和审阅者，代理负责实现细节。
