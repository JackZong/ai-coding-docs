# 幻觉与限制

> AI 模型并非全知全能。理解它的局限性，才能在编程中用对、用好。

## 什么是 AI 幻觉

**幻觉（Hallucination）** 是指 AI 模型生成了看似合理但实际上错误的内容。在编程场景中，这可能表现为：

- 编造不存在的 API 或库函数
- 生成语法正确但逻辑完全错误的代码
- 引用根本不存在的文档链接
- 自信满满地给出错误的技术解释

### 一个典型例子

你问模型："Python 的 `os.path.smart_join()` 函数怎么用？"

模型可能会一本正经地告诉你这个函数的参数和用法——然而 **`os.path.smart_join()` 根本不存在**。模型因为见过大量类似模式（如 `os.path.join`），"补全"出了一个看起来合理但虚构的答案。

## 幻觉产生的根本原因

回顾上一章的内容：LLM 的本质是**下一个 Token 的概率预测**。

这意味着：

| 模型在做的事 | 模型没有在做的事 |
|-------------|----------------|
| 根据统计概率生成文本 | 从数据库中查询事实 |
| 匹配训练数据中的模式 | 验证自己说的是否正确 |
| 生成"最可能"的下文 | 理解代码的运行时行为 |

模型没有"事实核查"机制。当它的训练数据中缺少某个答案时，它更倾向于**生成一个看似合理的虚构答案**，而不是说"我不知道"。

## AI 模型的核心限制

### 1. 知识截止（Knowledge Cutoff）

模型的知识来源于训练数据，有一个明确的时间截止点。截止日期之后发布的库、API 变更、新语法特性，模型一概不知。

> [!WARNING]
> 当你使用最新发布的框架或 API 时，模型生成的代码很可能基于旧版本，需要格外注意。

### 2. 无法执行和验证

模型本身无法运行代码。它不知道生成的代码是否能编译、是否能通过测试。它只是在"写"代码，而不是在"运行"代码。

### 3. 上下文依赖

模型只能基于你提供的上下文进行推理。如果项目中有自定义的工具函数、框架约定，模型无法自动感知——除非你明确告诉它。

### 4. 一致性问题

在长对话中，模型可能前后矛盾。比如前面说"用方案 A"，后面又改口建议"用方案 B"，却不会主动解释为什么改变了想法。

## 应对策略

### 1. 永远验证输出

**AI 的输出是"草稿"，不是"成品"**。每段生成的代码都应该：
- 实际运行并通过测试
- 经过人工代码审查
- 验证引用的 API/库确实存在

### 2. 提供充足的上下文

- 将相关的代码文件、类型定义、错误信息提供给模型
- 明确说明你使用的框架版本和技术栈
- 描述项目的约定和规范

### 3. 分步提问

不要一次性抛出复杂问题。将大任务分解为多个小步骤，每一步都验证后再继续。

```
❌ "帮我写一个完整的用户认证系统"
✅ "先帮我设计认证系统的数据模型，我确认后再写具体实现"
```

### 4. 交叉验证

- 让模型解释它生成的代码背后的逻辑
- 对关键决策，查阅官方文档来确认
- 对安全相关的代码，进行额外的专家审查

### 5. 利用工具调用

现代代码 Agent 可以通过工具调用来弥补 LLM 的局限——比如读取实际文件、执行代码、搜索文档。这大大减少了幻觉的可能性（在第 5 章"工具调用"中详细讨论）。

## 关键要点

- 幻觉是 LLM 的本质特征，而非偶尔的故障——因为它是在做概率预测，而非事实查询
- 了解模型的限制边界（知识截止、无法执行、上下文依赖），才能有效避免踩坑
- **对 AI 输出保持"信任但验证（Trust but Verify）"的态度**是最核心的原则
